# ADR-0003 · Bundled JSON with explicit versioning

**Status:** Accepted
**Date:** 2026-04
**Related:** `04-domain-model.md`, `05-crosswind-algorithm.md`, `09-cicd-and-ops.md`

## Context

Опорные значения для алгоритма расчёта Crosswind (slope, intercept-ы breakpoints, envelope границы) — это данные, которые могут меняться:

- При обновлении ревизии FCOM или OM-B.
- При уточнении операционной практики.
- При добавлении новых вариантов самолёта (B787-9) или состояний ВПП (Wet, Contaminated).

Возможные подходы хранения:

1. **Hardcoded в TypeScript-коде** — простейший, но любое изменение значения требует кода-PR и App Store update.
2. **Bundled JSON-файл** — данные отделены от кода, но обновление всё ещё требует пересборки приложения и (потенциально) App Store update.
3. **Удалённый источник** — JSON загружается с сервера; обновление мгновенное, но требует backend и онлайн-режима.
4. **Hybrid: bundled JSON + EAS Update** — данные в bundle, но при необходимости можно push-нуть OTA-обновление JSON через EAS Update без App Store re-submission.

Дополнительные требования:
- Приложение полностью офлайн (см. `01-vision.md`) — удалённый источник отвергается.
- Изменения данных должны быть прозрачны пользователю (отображение версии данных в About).
- Структура данных может эволюционировать (новые поля, новая модель интерполяции).
- Требуется runtime-валидация (на случай повреждения).

## Decision

**Используем bundled JSON-файлы с двухуровневым версионированием:**

- **`schemaVersion`** (semver) — версия структуры файла. Major bump при breaking-изменениях схемы.
- **`dataVersion`** (date-based) — версия конкретных значений. Инкрементируется при любом изменении значений.

Каждый JSON-файл соответствует фазе полёта; данные внутри ключуются по `(aircraft, runwayCondition)` (см. `04-domain-model.md` § "JSON Schema bundled данных"). В MVP — один файл `b787-takeoff.json` с заполненной только парой `byAircraft.b787_8.dry`. В Phase 2+ добавляются новые файлы (`b787-landing.json`) и/или новые ветки в существующих (B787-9, non-dry RWYCC).

Все JSON-файлы валидируются через zod-схему при загрузке репозиторием. При несоответствии — fail-safe режим, не показ числа.

В Phase 2+ опционально включается EAS Update для OTA-обновления значений без App Store re-submission, при условии что `schemaVersion` не меняется (структура совместима).

## Consequences

**Позитивные:**
- Data и code разделены — обновление значений не затрагивает архитектуру приложения.
- Версионирование делает прозрачным, какая версия данных в production.
- Runtime-валидация ловит повреждение данных до того, как приложение покажет неверный результат.
- В Phase 2+ возможны быстрые OTA-обновления через EAS Update.
- Легко добавлять новые комбинации (B787-9 → новый файл).

**Негативные:**
- Каждое изменение значений всё равно требует PR и тестирования (это намеренно — мы хотим, чтобы изменения проходили через quality gates).
- В MVP нет mechanism для изменения значений без App Store update — это appears как недостаток, но даёт reliability.
- zod-валидация добавляет ~50 ms к стартовой загрузке (приемлемо в performance budget).

**Нейтральные:**
- Существуют альтернативные библиотеки валидации (ajv, yup, io-ts) — zod выбран как лучший компромисс между type-safety и эргономикой в TypeScript.

## Alternatives considered

**Hardcoded в коде.** Отвергнут: нарушает разделение data/code, делает изменения значений более рискованными.

**Удалённый JSON через API.** Отвергнут: противоречит требованию полного офлайна. Также добавляет complexity (server, hosting, security).

**SQLite с миграциями.** Отвергнут: overkill для read-only справочных данных. Bundled JSON проще и достаточно.

**Encrypted bundled file.** Отвергнут: данные не секретные. Encryption добавит complexity без выгоды.
