# ADR-0001 · Clean Architecture only at module boundaries

**Status:** Accepted
**Date:** 2026-04
**Related:** `02-architecture.md`, `08-quality-gates.md`

## Context

Проект B787 Calculator — это приложение для пилотов с приоритетом на чистую архитектуру и долгосрочную расширяемость. Существует два полярных подхода:

- **«Strict Clean Architecture внутри каждого модуля»** — обязательный Use Case-класс для каждой публичной операции, обязательный Repository-интерфейс, обязательный mapper между data и domain слоями. Полный паттерн «Onion / Hexagonal architecture».
- **«Прагматичный Clean Architecture»** — чистые границы между модулями (presentation/domain/data), но внутри одного модуля минимум абстракций; Use Case появляется только когда логика реально координирует несколько источников.

Первый подход даёт максимальную защиту от технического долга и одинаковую структуру в каждом модуле, но удваивает количество файлов и заметно замедляет разработку первых модулей. Второй — быстрее и читабельнее в простых случаях, но требует дисциплины эскалации сложности.

Проект имеет одного MVP-модуля (Crosswind) с одним источником данных (bundled JSON) и одним вычислительным сценарием. Применение полного Clean Architecture ко всему модулю даст 5+ слоёв абстракции для одного простого расчёта.

## Decision

**Применяем Clean Architecture только на уровне границ между модулями.** Внутри одного feature-модуля сохраняется условное разделение на presentation / domain / data слои, но без принудительных Use Case-классов, Repository-интерфейсов или DTO-mapping-ов, когда они не дают выгоды.

Обязательные элементы:
- Изоляция модулей через barrel-экспорты и `no-restricted-paths` ESLint-правила.
- Domain-слой не зависит от React Native или Expo — чистый TypeScript.
- Repository pattern для данных (даже если реализация одна — bundled JSON).
- Чистые функции для расчётов.

Опциональные элементы (применяются по триггерам эскалации):
- Use Case-класс — когда логика координирует ≥ 2 источников данных или содержит сложные правила.
- Mapper между DTO и domain — когда структуры расходятся.
- Интерфейс Repository с in-memory fake — когда нужно тестировать в изоляции.

Триггеры эскалации задокументированы в `02-architecture.md`, раздел «Когда эскалировать архитектуру».

## Consequences

**Позитивные:**
- Быстрая разработка MVP без избыточного boilerplate.
- Читаемый код в простых случаях.
- Опытные разработчики не сталкиваются с «архитектурным шумом».
- Сложность добавляется тогда, когда она реально нужна, а не «на всякий случай».

**Негативные:**
- Требует дисциплины: разработчик и агент должны соблюдать триггеры эскалации, иначе код может deteriorate в простой mess.
- Code review должен явно проверять триггеры (это автоматизировано через ESLint complexity rules и max-lines).
- Структура кода в разных модулях может слегка отличаться (один с явным Use Case, другой без).

**Нейтральные:**
- При росте проекта потребуется ввод дополнительных абстракций; это рефакторинг, не катастрофа.

## Alternatives considered

**Strict Clean Architecture from day 1.** Отвергнут: для калькулятора это over-engineering. Удваивает время на каждый модуль без пропорциональной выгоды.

**No architecture, just file organization.** Отвергнут: создаст технический долг и сделает добавление новых модулей рискованным.

**MVC / MVVM only.** Отвергнут: не даёт достаточной модульности для расширения; смешивает domain и presentation.
