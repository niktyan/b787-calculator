# Спецификация B787 Calculator

Это исчерпывающий набор документов, описывающий проект **B787 Calculator** — iPad-приложение для пилотов Boeing 787, распространяемое в Public App Store.

Эта папка является **единственным источником правды** для проекта. Любой код, который противоречит документам в этой папке, является ошибкой. Любое архитектурное изменение должно сопровождаться обновлением соответствующего документа в том же Pull Request.

---

## Как читать документы

Документы пронумерованы в логическом порядке. Рекомендуется читать в следующей последовательности:

1. **`01-vision.md`** — что мы делаем, для кого и зачем. Читается первым.
2. **`02-architecture.md`** — как устроено приложение архитектурно.
3. **`03-tech-stack.md`** — на чём именно реализуется (точные библиотеки и версии).
4. **`04-domain-model.md`** — модели данных, JSON-схемы.
5. **`05-crosswind-algorithm.md`** — алгоритм расчёта Crosswind с тест-таблицей.
6. **`06-ui-spec.md`** — поведение каждого экрана и навигации.
7. **`07-app-store-compliance.md`** — тексты дисклеймеров, Privacy Manifest, App Store Listing.
8. **`08-quality-gates.md`** — TypeScript-правила, lint, тесты, перформанс, accessibility.
9. **`09-cicd-and-ops.md`** — GitHub-workflow, EAS Build, OTA-обновления, секреты, мониторинг.
10. **`10-glossary.md`** — авиационные и технические термины.

Дополнительно:

- **`AGENTS.md`** — инструкции для агента-реализатора (Claude Code) о том, как работать с проектом.
- **`ADR/`** — Architecture Decision Records: записи отдельных архитектурных решений с обоснованием.
- **`module-contracts/`** — публичные API каждого модуля. Описывают, что модуль предоставляет наружу и от чего зависит.

## Принципы спецификации

**Единственный источник правды.** Если документация и код расходятся — это баг. Документы обновляются в тех же PR, что и затронутый код.

**Каждый документ короткий и целевой.** 5–10 минут на чтение. Если документ растёт больше 1500 слов — он должен быть разбит.

**Решения с обоснованием.** Каждое нетривиальное решение сопровождается коротким объяснением «почему так, а не иначе» — это становится ADR.

**Никаких неподтверждённых предположений.** Если что-то не определено — явно указывается «open question», а не выдумывается на ходу.

**Read-once, build-many.** Документы пишутся так, чтобы агент-реализатор мог прочитать их и реализовать код без дополнительных вопросов в большинстве случаев.

## Использование с Claude Code

Агент-реализатор (Claude Code) читает эти документы автоматически при запуске сессии. Файл `CLAUDE.md` в корне репозитория ссылается на нужные документы и задаёт правила работы.

Для каждого спринта реализации существует подготовленный промпт, явно указывающий, какие документы агент обязан прочитать перед началом работы. Промпты лежат в `prompts/` (создаются по мере необходимости).

## Соглашения по тексту

- Язык документации — русский. Технические термины и идентификаторы — английский.
- Авиационные термины (KT, MAC, RWYCC, FCOM, OM-B и т.д.) пишутся как в оригинале.
- Имена файлов, переменных, классов в коде — `lowerCamelCase` или `PascalCase` согласно `08-quality-gates.md`.
- Числовые значения пишутся точно: `170 t`, `27.687 %MAC`, `40 KT`. Без округлений в спеке (округление происходит в коде).
